# Release 0.1.23 â€” let the gate speak once

Separate release publishing from the production CI workflow so version tags no longer replay the full verification suite or show skipped deploy jobs, while still refusing to publish a GitHub release until the matching `main` commit has verified and deployed.

## Highlights

- Keeps production deployment owned by the `main` push workflow.
- Moves tag-triggered GitHub release publishing into its own lightweight workflow.
- Requires release tags to point at commits contained in `main`.
- Waits for the matching `main` CI run to finish before publishing release notes.
- Requires both `verify` and `deploy-production` to succeed for the tagged commit.

## What changed

- Release metadata:
  - `package.json`
  - `docs/releases/README.md`
  - `docs/releases/0.1.23.md`
  - Bumps the package release line to `0.1.23` and adds this note to the release index.
- CI routing:
  - `.github/workflows/ci.yml`
  - Removes tag pushes from the main CI workflow trigger.
  - Leaves pull requests and `main` pushes on the full verify path.
  - Keeps `deploy-production` gated behind a successful `main` push verification.
- Release publishing:
  - `.github/workflows/release.yml`
  - Adds the tag-only release workflow.
  - Validates that release notes exist for the tag version.
  - Checks that the tag commit is on `main`.
  - Polls the matching `main` CI run and publishes only after `verify` and `deploy-production` complete successfully.

## Lore

Constraint: Release tags should describe a shipped commit without spending another full browser run or pretending a skipped deploy job is useful signal.
Rejected: Delete `deploy-production` | Vercel Git deploys are disabled in `vercel.json`, so removing the job would remove the production path rather than clean it.
Rejected: Let tag pushes publish immediately | That would make release notes faster, but it would allow a GitHub release before the matching `main` deploy proved itself.
Rejected: Keep release publishing inside `ci.yml` | The shared workflow made every release appear as two half-relevant runs, with one side always skipped.
Confidence: high
Scope-risk: narrow
Directive: Let `main` open the garden gate, then let the tag place the release marker after the path is clear.
Tested: `pnpm exec prettier --check .github/workflows/ci.yml .github/workflows/release.yml`; YAML parse for both workflows; `gh run list` / `gh run view` query against the current release commit, confirming `verify` and `deploy-production` succeeded; ShellCheck on extracted `run:` scripts from `release.yml`.
Not-tested: GitHub Actions production deploy and release publishing have not run for this tag yet.
