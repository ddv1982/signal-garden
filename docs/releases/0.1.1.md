# Release 0.1.1 — clear the deploy path

Harden the first production deploy workflow after the v0.1.0 `main` run exposed a slow visual-debug e2e gate.

## Highlights

- Keeps production deployment owned by the `main` push workflow.
- Stops release tags from launching duplicate CI runs that cannot deploy.
- Narrows the pet-debug e2e smoke so it verifies animation controls without replaying every full visual-debug sequence in one CI gate.

## What changed

- CI routing:
  - `.github/workflows/ci.yml`
  - Push-triggered CI now runs only for `main`, while pull requests still verify normally.
  - Tags remain release markers instead of separate deploy attempts.

- E2E deploy gate:
  - `e2e/garden-loop.spec.ts`
  - The pet-debug smoke still freezes every frame.
  - Sequence coverage now checks representative stretch, sleep, and wake flows with explicit start and final state assertions.
  - The separate sleep and reduced-motion tests continue to cover sequence behavior.

## Lore

Constraint: Keep e2e coverage in front of production deploys without letting a visual-debug review loop consume the deployment window.
Rejected: Deploy without e2e | The deploy workflow should still prove the browser path before Vercel receives the build.
Rejected: Move or retag `v0.1.0` | The first tag is already published; this follow-up records the deployment hardening honestly.
Confidence: high
Scope-risk: narrow
Directive: Let tags describe releases, but let `main` alone open the production gate.
Tested: `corepack pnpm run typecheck`; `corepack pnpm run test`; `corepack pnpm run e2e --project=chromium`; `corepack pnpm run build`.
Not-tested: The follow-up GitHub Actions production deploy has not run yet.
